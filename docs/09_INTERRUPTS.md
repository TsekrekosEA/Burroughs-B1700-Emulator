# 09 — Interrupt System

## 1. Overview

The B1700 uses a **soft interrupt system**. This means:

- Interrupt conditions do **NOT** cause any automatic hardware actions
- No interrupt vector table, no automatic register saving, no mode switch
- Interrupts are recognized **only** when software explicitly tests for them
- All interrupt handling is under complete **software/microcode control**

This design gives interpreters maximum flexibility in when and how interrupts
are processed, but it also means interrupt latency depends on how frequently
the interpreter polls for interrupt conditions.

---

## 2. Interrupt Sources

Four interrupt conditions are defined in hardware:

| # | Interrupt | Register Bit | Source |
|---|-----------|-------------|--------|
| 1 | **Timer Interrupt** | CC(2) | Real-time clock (every 100 ms) |
| 2 | **I/O Bus Service Request** | CC(1) | Any I/O control asserts SR |
| 3 | **Console Interrupt** | CC(0) | Operator presses console interrupt button |
| 4 | **Memory Parity Error** | CD(3) | Parity error on processor-initiated memory read |

### 2.1 Timer Interrupt

- Generated by the **Real-Time Clock** circuit on Card K
- Fires every **100 milliseconds**
- Derived from AC power frequency (60 Hz or 50 Hz, adjustable by field engineer)
- Sets **CC(2)** in the C register
- Primary use: preemptive time-slicing for multiprogramming

### 2.2 I/O Bus Service Request

- Any I/O control can assert a **Service Request (SR)** signal on the I/O bus
- Sets **CC(1)** in the C register
- Indicates an I/O operation completed, device needs attention, or error occurred
- Software must poll the specific I/O control to determine the cause

### 2.3 Console Interrupt

- Triggered by the operator pressing the **Interrupt button** on the console
- Sets **CC(0)** in the C register
- Used for operator intervention (halt programs, enter commands, etc.)

### 2.4 Memory Parity Error

- Set when a **parity error** is detected during a memory read cycle
  initiated by the processor
- Sets **CD(3)** in the C register
- Indicates a hardware memory fault

## 3. Interrupt Detection

### 3.1 The INT OR Signal

The **XYST** pseudo-register (Group 12/Select 3) contains a bit called
**INT OR** at position XYST(2). This is the logical OR of all interrupt
conditions:

```
XYST(2) = CC(0) OR CC(1) OR CC(2) OR CD(3)
```

**INT OR** provides a fast single-bit check: if it's false, no interrupts
are pending and no further checking is needed.

### 3.2 Typical Polling Sequence

The interpreter's main loop includes an interrupt check:

```
; At end of each S-instruction execution:

; Step 1: Fast check — any interrupt at all?
6C: SKIP WHEN XYST(2) = 1    ; test INT OR bit
    ; (if not set, skip the interrupt handler — fall through to next S-instruction)

; Step 2: If INT OR set, determine which interrupt(s)
4C: BIT TEST CC(2) BRANCH_FALSE skip_timer
    ; Timer interrupt — handle it
    14C: CALL timer_handler
skip_timer:

4C: BIT TEST CC(1) BRANCH_FALSE skip_io
    ; I/O interrupt — handle it
    14C: CALL io_handler
skip_io:

4C: BIT TEST CC(0) BRANCH_FALSE skip_console
    ; Console interrupt — handle it
    14C: CALL console_handler
skip_console:

4C: BIT TEST CD(3) BRANCH_FALSE skip_parity
    ; Memory parity error — handle it
    14C: CALL parity_handler
skip_parity:
```

### 3.3 Interrupt Priority

Since interrupts are entirely software-polled, the **interpreter determines
priority** by the order in which it tests the flags. There is no hardware
priority encoder.

Typical priority order (by convention):
1. Memory parity error (highest — hardware fault)
2. Timer interrupt (scheduling)
3. I/O service request (device completion)
4. Console interrupt (operator, lowest)

### 3.4 Clearing Interrupt Flags

Interrupt flags in CC and CD are **read/write** — software can clear them
by writing zeros:

```
; Clear timer interrupt flag
3C: CC AND 1011 → CC    ; clear bit 2, preserve bits 0,1,3
```

Or more simply:
```
; Set CC to specific value
1C: MOVE literal → CC
```

The interrupt source hardware may re-set the flag if the condition persists
(e.g., an I/O control still asserting SR will keep CC(1) high).

## 4. Interrupt Handling Flow

```
┌───────────────────────────────────────────────┐
│         Normal S-instruction Execution         │
│                                                │
│  Execute S-instruction                         │
│  ↓                                             │
│  Test XYST(2) — INT OR                        │
│  ├─ False → Loop to next S-instruction        │
│  └─ True ↓                                     │
│     Test individual flags (CC, CD)             │
│     ├─ Timer? → Save context, call MCP         │
│     │    MCP may: reschedule, swap programs     │
│     ├─ I/O? → Identify device, process status  │
│     ├─ Console? → Process operator input       │
│     └─ Parity? → Error recovery/abort          │
│     Return to normal execution                 │
└───────────────────────────────────────────────┘
```

---

## 5. Timer Interrupt and Scheduling

The 100 ms timer interrupt is the mechanism for **preemptive multitasking**:

1. Timer fires → CC(2) set
2. Interpreter detects INT OR during polling
3. Interpreter saves current program state to memory
4. Interpreter returns control to MCP
5. MCP evaluates job priorities and decides whether to continue current
   program or switch to another
6. If switching: new interpreter loaded (if needed), new program context
   restored
7. Execution resumes

## 6. C++ Implementation Notes

### 6.1 Interrupt State

```cpp
struct InterruptState {
    // CC register bits (also stored in RegisterFile::CC)
    // CC(0) = console interrupt
    // CC(1) = I/O bus service request
    // CC(2) = timer interrupt
    // CC(3) = console state lamp (not an interrupt)

    // CD register bits
    // CD(3) = memory parity error

    // Derived: INT OR
    bool int_or() const {
        return (regs.CC & 0x07) != 0 ||  // CC bits 0-2
               (regs.CD & 0x08) != 0;    // CD bit 3
    }

    // Timer management
    uint64_t timer_cycle_count = 0;
    uint64_t cycles_per_100ms;   // computed from clock frequency

    void tick_timer(uint64_t current_cycle) {
        if (current_cycle - timer_cycle_count >= cycles_per_100ms) {
            timer_cycle_count = current_cycle;
            regs.CC |= 0x04;  // set CC(2) timer interrupt
        }
    }
};
```

### 6.2 Key Points

1. **No hardware interrupt vectoring**: The emulator does NOT need to implement
   an interrupt dispatch mechanism. It only needs to:
   - Set CC/CD bits when conditions occur
   - Maintain XYST (including INT OR) as a combinatorial output
   - Let the software do the rest

2. **Timer accuracy**: The 100 ms timer should be derived from emulated cycle
   counts, not real wall-clock time, for deterministic behavior. However,
   real-time mode can use wall-clock for interactive use.

3. **I/O interrupts**: Set CC(1) whenever any IOControl reports a service
   request.

4. **XYST is read-only**: The emulator computes XYST(2) on-the-fly from
   CC and CD values.
